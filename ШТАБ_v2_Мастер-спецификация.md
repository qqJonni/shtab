# ШТАБ v2 — Мастер-спецификация (чистый проект)

Новый проект, пишется с нуля. Из старой версии переносим **дизайн-язык** (палитра, sidebar/topbar/мобильное меню, компоненты — см. раздел 9) и проверенный стек. Всё остальное проектируем заново под полный цикл строительства и документооборот.

> Цель продукта: единая платформа управления строительством полного цикла для девелопера — от заведения объекта и этапов до закрытия работ, согласования первичных документов (КС-2/КС-3 и др.) и снабжения. Первый клиент — ГК «Федерация» (Пермь). Ориентир — PlanRadar, но дешевле, в РФ, нативно на языке российских документов и под реальные процессы клиента.

---

## 1. Подход
- Разработка локальная, модулями. Для каждого модуля — отдельное ТЗ с критериями приёмки.
- Код пишет Claude Code с чистого листа. Я (стратег) даю спецификацию и пошаговые ТЗ, делаю ревью.
- Каждый модуль должен запускаться и не ломать предыдущие. Менять по одному модулю за раз.

## 2. Стек и принципы (переносим проверенное)
- Backend: **Flask** (app-factory) + **Flask-Login**.
- БД: **SQLite** (один файл) — на этапе локальной разработки и пилота достаточно. Архитектуру данных проектируем так, чтобы при необходимости переехать на Postgres (явные FK, без SQLite-специфики в логике).
- Frontend: **Bootstrap 5.3.3 + Bootstrap Icons + шрифт Inter**, кастом в `static/css/crm.css`.
- Excel: **openpyxl** (экспорт сводных + генерация форм КС). PDF — позже при необходимости.
- Чистая модульная структура с самого начала (blueprints/модули, слой БД, RBAC-декораторы, дизайн-система).
- Уведомления — **внутренний центр уведомлений** (таблица + колокольчик со счётчиком). Настоящий web/email-push — отдельный модуль позже.

## 3. Роли
Бизнес-роли (7) + системные (2).

| Роль (код) | Название | Суть |
|---|---|---|
| `manager` | Руководитель | Заводит объекты и этапы, ищет/назначает подрядчика, загружает документы этапа, финальный согласующий в цепочке, общий контроль, дашборды |
| `pto` | Инженер ПТО | Формирует/редактирует подэтапы по ценовому документу, ведёт расценки, проверяет заявки на материал, согласующий в цепочке |
| `inspector` | Технадзор | Проверяет ход всех этапов/подэтапов, выдаёт замечания, **первый** согласующий в цепочке документов |
| `foreman` | Прораб | Контроль на площадке, выдаёт замечания подрядчикам, **второй** согласующий |
| `supply` | Снабженец | Обрабатывает заявки на давальческий материал |
| `accountant` | Бухгалтер | Финальный получатель согласованного пакета документов |
| `contractor` | Подрядчик | Видит свои этапы/подэтапы, грузит документы, закрывает подэтапы, формирует КС-2/КС-3/счёт-фактуру/отчёт по давальческому, отправляет в цепочку, делает заявки на материал, устраняет замечания |
| `admin` | Администратор | Системная роль: пользователи, организации, настройки. Может действовать как суперпользователь |
| `guest` | Гость | Просмотр только по токен-ссылке |

> Решение к подтверждению: ты перечислил 7 бизнес-ролей. `admin` оставляю как системного суперпользователя (управление пользователями/настройками неизбежно нужно), `guest` — ради сохраняемой фишки гостевых ссылок. Если хочешь, чтобы функции admin выполнял `manager` — скажи, уберём отдельную роль.

Доступ — строго через декоратор `@role_required(...)`, не хардкодом в шаблонах.

## 4. Доменная модель

### 4.1 Организации и пользователи
```
organizations: id, name, type('developer'|'contractor'), inn, kpp, created_at
users: id, username, password_hash, role, full_name, avatar,
       is_approved, organization_id→organizations, created_at
```
Подрядчик-пользователи привязаны к организации-подрядчику.

### 4.2 Объект и этапы
```
objects: id, name, address, type, status('active'|'archived'),
         created_by→users, created_at

construction_stages (Этап строительства):
  id, object_id→objects, name, order_num, description,
  contractor_id→organizations (NULL пока не назначен),
  contractor_status('search'|'assigned'),
  plan_start_date, plan_end_date,        -- договорные сроки
  status('planned'|'in_progress'|'done'|'suspended'),
  created_by→users, created_at

stage_documents (Документы этапа: договор, ТЗ, график производства работ, прочее):
  id, stage_id→construction_stages,
  doc_type('contract'|'tech_spec'|'work_schedule'|'other'),
  title, filename, uploaded_by→users, uploaded_at
```
Этапы заводит **руководитель**, он же назначает подрядчика и грузит документы этапа. Подрядчик видит свои этапы и может грузить/смотреть/удалять документы этапа.

### 4.3 Подэтапы (состав работ по ценовому документу)
```
substages (Подэтап):
  id, stage_id→construction_stages, name, description,
  volume REAL, unit, unit_price REAL,      -- расценка из ценового документа
  total_price REAL,                        -- volume*unit_price (хранимое/вычисляемое)
  plan_end_date,                           -- срок ≤ срок этапа
  status('not_started'|'in_progress'|'done'|'closed'|'approved'),
  created_by→users, created_at, completed_at

substage_photos: id, substage_id→substages, filename, uploaded_by, uploaded_at
```
Подэтапы формирует/редактирует **ПТО** (и при необходимости руководитель). Подрядчик наследуется от родительского этапа. Срок подэтапа не превышает срок этапа (валидация). `closed` = подрядчик закрыл подэтап и отправил пакет документов; `approved` = пакет прошёл всю цепочку.

### 4.4 Закрытие подэтапа и маршрут согласования документов (ядро)
```
doc_packages (Пакет закрытия подэтапа):
  id, substage_id→substages, contractor_id→organizations, created_by→users,
  status('draft'|'in_review'|'returned'|'approved'|'completed'),
  return_to_role,                          -- куда вернётся после правок подрядчика
  created_at, submitted_at, completed_at

package_documents (Документы пакета):
  id, package_id→doc_packages,
  doc_type('ks2'|'ks3'|'invoice'|'raw_material_report'|'free_form'),
  title, filename, is_generated INTEGER,   -- сгенерирован системой или загружен
  created_at

approval_steps (Шаги маршрута согласования):
  id, package_id→doc_packages, step_order INTEGER,
  role('inspector'|'foreman'|'pto'|'manager'|'accountant'),
  status('pending'|'approved'|'returned'|'waiting'),
  approver_id→users, comment, acted_at
```
Формы КС-2, КС-3, счёт-фактура, отчёт о расходе давальческих материалов — **стандартные** (генерируем по бланкам клиента, бланки приложит Лео). Прочие документы — в свободной форме (загрузка файлом).

### 4.5 Замечания
```
defect_types: id, name, order_num
defects:
  id, object_id→objects, stage_id→construction_stages, substage_id→substages (NULL),
  title, description, type_id→defect_types,
  priority('low'|'normal'|'high'|'critical'),
  status('open'|'in_progress'|'resolved'|'verified'|'rejected'|'closed'),
  reporter_id→users, responsible_id→users, contractor_id→organizations,
  due_date, created_at, updated_at, resolved_at, verified_at, reopen_count
defect_photos: id, defect_id, filename, photo_type('before'|'after'|'general'), uploaded_by, uploaded_at
defect_history: id, defect_id, user_id, action, old_value, new_value, comment, created_at
```
Выдают **технадзор и прораб** → подрядчику. (Задел на будущее: привязка замечания к точке на плане/BIM — отдельный модуль.)

### 4.6 Снабжение (давальческий материал)
```
material_requests (Заявка на давальческий материал):
  id, stage_id→construction_stages, substage_id→substages (NULL),
  contractor_id→organizations, requested_by→users,
  status('submitted'|'returned'|'approved'|'processing'|'completed'),
  current_role('pto'|'supply'),
  created_at, updated_at, completed_at
material_request_items:
  id, request_id→material_requests, material_name, unit, quantity REAL, price REAL, comment
material_request_history:
  id, request_id, user_id, action, comment, created_at
```
Заявка — по стандартной форме, позиции — согласно ценовому документу.

### 4.7 Уведомления
```
notifications: id, user_id→users (получатель), type, title, body, link, is_read, created_at
```
Колокольчик со счётчиком непрочитанных в топбаре + страница-лента. Создаются хелпером `notify(user_id, type, title, body, link)` из всех воркфлоу.

### 4.8 Настройки и гостевые токены
```
settings: id, key UNIQUE, value
guest_tokens: id, object_id→objects, token UNIQUE, created_by, created_at
```

## 5. Воркфлоу

### 5.1 Объект → этапы → подрядчик → документы этапа
1. Руководитель заводит **объект** (МКД).
2. Заводит **этапы строительства** в произвольном составе (котлован, свайное поле, фундамент, монолитный каркас, …), задаёт порядок и плановые/договорные сроки.
3. По этапу идёт **поиск подрядчика** (вне приложения; состав работ этапа можно выгрузить документом). Руководитель отдельной ролью «тендер» не выделяется.
4. Найденного подрядчика руководитель **назначает на этап** (выбор организации-подрядчика), `contractor_status='assigned'`.
5. К этапу прикрепляются **документы**: договор, ТЗ, график производства работ и прочее (загрузка/просмотр/удаление). Доступ — руководитель и назначенный подрядчик.

### 5.2 Подэтапы
1. После назначения подрядчика **ПТО** (или руководитель) разбивает этап на **подэтапы** по согласованному ценовому документу (объём, ед.изм., расценка).
2. Каждому подэтапу — свой **срок ≤ срока этапа**.
3. **Подрядчик** видит свой этап, все подэтапы (состав), грузит/смотрит/удаляет документы, ведёт статус выполнения, грузит фото.

### 5.3 Замечания
В любой момент технадзор/прораб создаёт замечание подрядчику: текст + фото → подрядчику уведомление. Цикл:
`open → in_progress (подрядчик взял) → resolved (подрядчик: фото «после» + комментарий) → verified/closed (технадзор проверил)`; неустранено → возврат (`reopen`, причина, счётчик). На каждом переходе — уведомление соответствующей стороне.

### 5.4 Закрытие подэтапа и согласование документов (центр продукта)
Подрядчик завершает подэтап → собирает пакет документов (КС-2, КС-3, счёт-фактура, отчёт по давальческому, плюс свободные) → **отправляет**. Пакет идёт по фиксированной цепочке; на каждом шаге согласующий **принимает** или **возвращает подрядчику с комментарием**.

```
Подрядчик закрыл подэтап и отправил пакет
        │  (уведомление → Технадзор)
        ▼
[1] Технадзор  ──принял──►[2] Прораб ──принял──►[3] ПТО ──принял──►[4] Руководитель ──принял──►[5] Бухгалтерия (готово)
        │                       │                    │                     │
        └──вернул подрядчику────┴────────────────────┴─────────────────────┘
                с комментарием (например «объёмы не совпадают»)
```
Правила:
- Старт цепочки — **технадзор** (ему первое уведомление: «подрядчик X закрывает этап Y»). Он проверяет, что заявленный объём реально выполнен и качественно, ставит согласование.
- Далее по очереди: прораб → ПТО → руководитель → бухгалтерия. Каждому — уведомление, когда очередь дошла.
- **Возврат**: любой согласующий может вернуть пакет подрядчику с комментарием. Ранее проставленные согласования **сохраняются**.
- **Прямой возврат**: после правок/ответа подрядчик возвращает пакет **сразу той роли, что вернула** (не проходит всю цепочку заново). Реализуется через `return_to_role` / реактивацию конкретного шага.
- **Видимость согласований**: каждый следующий видит, кто уже согласовал. Пример: ПТО видит, что согласовали технадзор и прораб; бухгалтер — что согласовали технадзор, прораб, ПТО, руководитель.
- Формы КС-2/КС-3/счёт-фактура/отчёт по давальческому — по стандартным бланкам клиента (генерация), прочее — свободная форма (загрузка).

### 5.5 Снабжение
Подрядчик создаёт **заявку на давальческий материал** (стандартная форма, позиции по ценовому документу) → уведомление **ПТО**. ПТО проверяет: **вернуть с комментарием** или **подтвердить** → уведомление **снабженцу**. Снабженец обрабатывает, **завершает** заявку → уходит в архив.

### 5.6 Уведомления (триггеры)
Подэтап закрыт → технадзору. Замечание создано → подрядчику. Замечание устранено → технадзору. Очередь согласования перешла → следующей роли. Пакет/заявка возвращены → подрядчику. Заявка подтверждена ПТО → снабженцу.

## 8. Разбивка на модули (порядок сборки)
- **Модуль 0 — Каркас.** App-factory, config, слой БД, структура модулей, авторизация (вход/регистрация/роли), `@role_required`, базовый layout + `crm.css` (перенос дизайна), навигация-скелет по всем ролям, инфраструктура уведомлений (таблица + хелпер + колокольчик), seed-пользователи всех ролей.
- **Модуль 1 — Объекты и этапы.** CRUD объектов (руководитель), CRUD этапов (порядок, сроки), назначение подрядчика, документы этапа (договор/ТЗ/график). Подрядчик видит свои этапы.
- **Модуль 2 — Подэтапы.** CRUD подэтапов (ПТО/руководитель) с объёмом/ед./расценкой и сроком ≤ этап; ценовой документ; вид подрядчика (состав этапа), документы и фото подэтапа.
- **Модуль 3 — Замечания.** Полный цикл, фото до/после, история, уведомления, привязка к этапу/подэтапу.
- **Модуль 4 — Закрытие подэтапа и маршрут согласования.** Пакеты, документы пакета (свободная форма + генерация КС когда будут бланки), цепочка согласования с возвратом и прямым возвратом, видимость согласований, уведомления. **Ядро.**
- **Модуль 5 — Снабжение.** Заявки на давальческий материал, цепочка подрядчик→ПТО→снабженец→архив, уведомления.
- **Модуль 6 — Дашборды, отчётность, экспорт, гостевые ссылки.** Дашборд руководителя (диаграммы/сводки), ролевые дашборды, Excel-экспорт, гостевые токены.

## 9. Открытые вопросы и зависимости
- **Бланки документов**: для генерации КС-2, КС-3, счёта-фактуры, отчёта о расходе давальческих материалов нужны реальные шаблоны клиента. До их получения Модуль 4 делает воркфлоу + загрузку файлами; генерация форм подключается, когда бланки будут.
- **Роли admin/guest** — подтвердить (раздел 3).
- **Push** — в этой версии центр уведомлений в приложении; настоящий web/email-push вынесен в отдельный будущий модуль.
- **BIM/пины на плане** — будущий модуль (как доп. фишка к замечаниям).
